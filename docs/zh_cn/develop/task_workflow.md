# 任务流程采集与开发规范

本文规定 MaaFramework 任务在实现前的流程采集方式，以及实现后的验证要求。目标是让 Pipeline 基于真实页面状态设计，同时避免为小改动制造不必要的文档负担。

## 变更分级

### 新任务或主流程重构

新增完整任务、调整跨页面路径或改变任务结束条件时，开发前必须在模拟器中至少观察一次可执行的主流程，并在 `docs/zh_cn/task_flows/` 新建对应记录，例如 `docs/zh_cn/task_flows/task_daily_shop.md`。一次运行不代表覆盖全部情况；还需根据可见页面和现有行为列出已知异常分支。

### 局部行为修改

修改单个 ROI、OCR 文本、模板、按钮或异常分支时，只需复现受影响页面并验证对应子流程。修改前现象、截图或日志及修改后结果可直接记录在 Issue 或 PR，无需重写完整任务文档。

### 非行为修改

纯文档、格式化、构建配置及不改变运行结果的重构无需启动模拟器，但仍须执行适用的静态检查。

## 迁移文档审查

从其他项目、旧分支或个人笔记搬入的内容不能直接视为本项目规范或验证证据。整理时必须核对当前仓库的目录、Interface 任务与 Option、Pipeline 节点、Agent 注册名、工具命令和客户端页面。不存在的路径或工具应改为本仓库实现；无法确认的设计说明应删除或标为待验证。旧文档中的“已实测”“已修复”只有在能确认对应版本、设备和结果仍适用于当前 TKFMaa 时才能保留为有效验证，否则仅记录为迁移线索。

## 标准开发步骤

1. 明确任务入口、前置状态、资源条件和期望结束状态。
2. 在模拟器中按真实操作运行流程，记录每次页面变化及可稳定识别的特征。
3. 补充加载、弹窗、资源不足、目标不存在和返回失败等可预见分支。
4. 根据记录设计 `next`/`[JumpBack]` 状态机，再编写 Pipeline；复杂运行时决策才交给 Python Agent。
5. 先验证独立识别节点和动作，再运行完整任务。
6. 执行 `maa-tools check`、JSON Schema 校验，并在 PR 中写明实机验证范围和结果。

开始实机采集前，在 [`task_capture_plan.md`](./task_capture_plan.md) 选择一个任务，并复制其中的单任务采集表。只有实际保存截图并核对页面后才能更新采集状态；现有 Pipeline、旧文档或推测流程不能计为已采集。

## 生成配置同步

体力活动采用源配置与派生产物分离的方式：维护者编辑 `assets/stamina_activities.yaml`，再由 `tools/build_stamina_activities.py` 更新 `assets/interface.json` 中“体力消耗关卡”的 `default_case` 和 `cases`。不要直接编辑这些生成字段，否则下次运行脚本时会被覆盖。

按以下顺序操作：

1. 运行 `python tools/build_stamina_activities.py --check` 判断当前是否同步。
2. 修改 YAML 或影响生成结果的脚本逻辑后，运行 `python tools/build_stamina_activities.py`。
3. 再次运行 `python tools/build_stamina_activities.py --check`，必须得到同步成功结果。
4. 审查 `assets/interface.json` 差异，确认只包含预期 Case、默认项和 Override 变化。
5. 执行 Prettier、`maa-tools check`、JSON Schema 校验和受影响路线的实机验证。

源配置、生成脚本和对应派生产物必须在同一提交中保持可复现。只修改无关任务或文档时无需运行生成命令；无法判断时运行 `--check`，不要为了“保险”无条件重写文件。

## 流程记录模板

```markdown
# <任务名称>

## 前置状态

- 当前页面：
- 账号或资源条件：

## 主流程

1. <当前页面、执行操作、目标页面>

## 页面识别

| 页面状态 | 识别目标 | 识别方式                 | ROI/素材 |
| -------- | -------- | ------------------------ | -------- |
|          |          | OCR/TemplateMatch/Custom |          |

## 异常与分支

- <触发条件>：<预期处理及恢复页面>

## 完成状态

- 成功条件：
- 安全退出位置：

## 验证记录

- 日期、客户端版本及设备：
- 已覆盖路径：
- 未覆盖风险：
```

流程记录只描述影响自动化设计的事实，不必写成面向玩家的操作教程。截图仅在无法准确描述页面或识别区域时添加，并避免提交包含账号隐私的信息。

## 验收要求

普通任务至少验证主流程、可恢复的常见弹窗和安全退出路径。购买、资源消耗、招募等高风险任务必须确认数量、按钮目标及终止条件，并完成一次端到端实机验证。暂时无法覆盖的分支必须在流程记录和 PR 中明确标注，不得以推测结果冒充已验证行为。
